# 🚀 스마트 방문 인증 (Fast Check-in) 업데이트 플랜

## 1. 개요 및 목적
현재 사용자가 수동으로 화장실 아이콘을 누르고 '방문 인증'을 해야만 1분 체류 시간 카운트가 시작되는 구조입니다. 이를 개선하여 **"앱(웹)을 실행하고 화장실 150m 이내에 진입한 순간"부터 자동으로 체류 시간을 기록**하는 아키텍처로 개편합니다.
이를 통해 실제 볼일을 마친 유저가 대기 시간 없이 즉시 **AI 촬영 인증** 절차로 넘어갈 수 있도록 사용자 경험(UX)을 극적으로 향상시키는 것이 목표입니다.

---

## 2. 주요 기술 원리 (Technical Principles)
- **Silent Ping (백그라운드 통신)**: 프론트엔드 애플리케이션 화면이 활성화되어 있는 동안(Foreground), 유저 몰래 백그라운드에서 주기적으로 위치를 확인하여 백엔드에 방문 정보를 전달합니다.
- **Set If Absent (멱등성 보장)**: 백엔드의 Redis는 동일한 체크인 요청이 여러 번 들어와도 **가장 처음에 등록된 시간(최초 도착 시간)**만 보존하도록 설계하여 다중 요청에 대비합니다.

---

## 3. 프론트엔드(Frontend) 업데이트 상세

프론트엔드는 유저의 위치 변화를 자동으로 감지하고 처리하는 "옵저버(Observer)" 역할을 수행합니다.

### 📍 [작업 1] GeoLocation Watcher (자동 위치 추적기) 구현
- 앱 메인에 진입하여 위치 권한이 승인된 시점부터 `navigator.geolocation.watchPosition` API를 호출합니다.
- 사용자의 실시간 좌표(위도, 경도) 상태를 10~20초 주기로 전역 컨텍스트(또는 Zustand)에 업데이트합니다.

### 📡 [작업 2] Silent Check-in (백그라운드 자동 호출) 도입
- 가져온 실시간 좌표와 로드된 가장 가까운 화장실 좌표 간의 거리를 계산(`haversine` 공식 등 활용)합니다.
- 특정 화장실과 **150m 이내(반경 이내)**에 진입한 것이 감지되면, 유저의 명시적 동작 없이도 `POST /api/v1/records/check-in` API를 호출(Silent Ping)합니다.
- 해당 호출이 반복되어 서버에 무리를 주지 않도록, 특정 화장실에 대해 핑을 보낸 후에는 **1~2분간 동일한 요청을 보내지 않는 쓰로틀링(Throttling) 방어 코드**를 구현합니다.

### ⏱️ [작업 3] 동적 대기 시간 UI/UX 개선
- 유저가 실제 볼일을 마치고 방문할 화장실 핀을 눌러 **[방문인증]** 모달(현재의 `VisitModal.tsx`)을 오픈합니다.
- 이때 이미 작업 2에 의해 백그라운드 체크인이 진행되었으므로,
  - **1분이 이미 지났을 경우**: 대기 시간 연출(타이머)을 아예 생략하고 즉시 "AI 기록 생성(사진 업로드 폼)" 모드가 활성화되도록 분기 처리합니다.
  - **예를 들어 40초만 지났을 경우**: 남은 시간 "20초"부터 타이머가 시작되도록 시각적 피드백을 제공합니다.

---

## 4. 백엔드(Backend) 업데이트 상세

백엔드는 기존의 로직을 유지하되, 프론트엔드가 남은 대기 시간을 계산하기 쉽도록 데이터 응답 스펙을 일부 개선합니다.

### 🔒 [작업 1] 체류 시간 기록 로직 안전성 검증
- 현재 `LocationVerificationService.java` 내의 `recordArrivalTime` 메서드에서 사용되는 Redis `setIfAbsent` 로직을 그대로 유지합니다. 이를 통해 프론트엔드에서 핑이 여러 번 오더라도 **최초 도착 시간**만 안전하게 1시간 동안 보장됩니다.
- 이 부분을 주석 또는 문서에 "Fast Check-in을 위한 최초 도착 시간 보존"이라고 명확히 로깅하고 명세화합니다.

### ⏳ [작업 2] Check-in API 응답(Response) 확장 
- 현재 `POST /api/v1/records/check-in`의 응답은 단순 `200 OK (Void)`입니다.
- 프론트엔드가 타이머를 정확히 동기화할 수 있도록, 이 응답을 다음과 같은 JSON 형태로 변경(Update)하는 로직을 추가합니다.
```json
{
  "toiletId": 38,
  "firstArrivalTime": "2026-03-23T18:00:00", // 서버에 기록된 최초 도착 시간
  "elapsedSeconds": 45, // 현재 시점 기준 경과된 시간
  "remainedSeconds": 15 // 1분까지 남은 시간
}
```
- 프론트엔드는 이 응답(`remainedSeconds`)을 기반으로 타이머 UI를 정확히 맞춰 렌더링할 수 있습니다.

### 🛡️ [작업 3] 어뷰징 로깅 (Optional)
- 정상적인 속도로는 불가능한 좌표 이동(예: 강남역에서 즉시 부산역으로 150m 진입 핑 발생 등)이 짧은 시간에 다발적으로 감지될 경우를 대비해, 위치 급변 어뷰징 방지 스프링 인터셉터 또는 로직을 고도화할 수 있습니다.

## 5. 단계별 실행 계획 (Roadmap)
1. **[Backend]**: `PooRecordController` 및 `PooCheckInResponse` DTO 생성을 통한 응답 포맷 개선.
2. **[Backend]**: 기능 작동 검증(Test Code 보완) 및 프론트엔드에 API 명세 전달.
3. **[Frontend]**: 기존 `useEffect` 기반 위치 추적 코드를 `watchPosition`으로 고도화하고 자동 Silent Ping 로직 (`Axios`) 추가.
4. **[Frontend]**: `VisitModal.tsx` 내 체류 시간 타이머 연동 및 대기 시간 Zero UI 분기 로직 개발.
5. **통합 테스트**: 실제 앱 실행 후 도보 이동 시나리오 테스트.
